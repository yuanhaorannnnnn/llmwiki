TL;DR

AirSim 的自研 FastPhysics 无人机动力学模型无法满足所有场景。 维护者 jonyMarino 发起讨论后，社区偏好明确转向 JSBSim（库级编译集成）， 否决了 Gazebo（运行时集成有性能代价）。 106 天后 PR #3626 合并，实现了外部动力学模拟器接口，让用户可替换物理引擎。

结论与决议

✅
外部动力学接口方案被采纳，替代了"改进 FastPhysics"的思路
PR #3626 实现了通过 setVehiclePose() 从外部模拟器注入位姿的接口。保留 FastPhysics 现有能力的同时，让用户可自由选择外部物理引擎。
✅
JSBSim 成为事实参考实现
jonyMarino 从最初就偏向 JSBSim（编译时集成、更精确的物理模型）；AOS55 的调研和 Fixedwing-Airsim 原型进一步验证了可行性。Gazebo 未进入实际集成路线。
✅
澄清了 AirSim 物理架构
FastPhysics（无人机/四旋翼）≠ PhysX（汽车）≠ Chaos（未使用）。关键认知：AirSim 的物理引擎选择因车辆类型而异。

未解决的问题

  • 原生 C++ JSBSim 集成（#2955）的最终目标尚未实现——AOS55 的 Fixedwing-Airsim 仍基于 Python 客户端方案
  • Chrono/synChrono 和 CARLA 被提及但未经过深入评估
  • Gazebo 的性能/架构担忧未被充分验证和反驳——是合理否决还是路径选择，无法从讨论中判断
